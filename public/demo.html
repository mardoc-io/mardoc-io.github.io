1. Executive summary

What OTel is
A vendor-neutral standard for collecting traces, metrics, and logs from applications.
Core idea
A single traceparent header flows through every service; a Collector aggregates the data.
Our opportunity
We can receive OTel data as an additional exporter — we do not need to replace anything.

OpenTelemetry (OTel) is a CNCF-graduated open-source standard that gives every participating application a common way to emit telemetry — traces, metrics, and logs. A W3C Trace Context header (traceparent) carries a trace ID across service boundaries so that downstream systems can correlate all the spans belonging to a single request. The OTel Collector sits between applications and observability backends, receiving, processing, and exporting telemetry. As a FinOps integrator, we can tap into this pipeline without disrupting existing observability setups.

2. What is OpenTelemetry?

OpenTelemetry is an instrumentation framework — it standardizes how applications generate and transmit telemetry data. It is not a storage or visualization backend (like Datadog, Grafana, or AWS X-Ray). Instead, it provides SDKs that applications embed, and a Collector process that routes the data to one or more backends of your choice.

OTel covers three "signal types": traces (a request's journey across services), metrics (numeric measurements like latency or error rates), and logs (structured event records). For our purposes, traces are the primary concern because they carry the correlation ID that ties an entire request lifecycle together.

3. W3C Trace Context & how correlation works

The W3C Trace Context specification defines an HTTP header called traceparent that propagates tracing identity across service boundaries. Its format is:

traceparent: 00-<trace-id>-<parent-span-id>-<trace-flags>

The trace ID (32 hex characters) is the correlation key. It is generated by the first service that handles a request and propagated — unchanged — to every downstream service. Each service creates its own span (with a unique 16-character span ID) and attaches that span to the shared trace ID. The Collector (or backend) then reconstructs the full call tree by grouping spans that share the same trace ID.

Key insight: The trace ID is the universal correlation key. Any system that receives spans tagged with the same trace ID can reconstruct the request's full journey. This is exactly how we, as an integrator, can correlate cost data with application request paths.

4. Spans, events, and custom attributes

A span represents a single unit of work — an HTTP handler, a database query, a message consumer. Every span contains a start time, an end time, a name, and a status. Beyond these basics, there are two mechanisms for attaching custom data to a span:

Span attributes

Key-value pairs set on the span itself. These describe the work being done: http.method = "POST", db.system = "postgresql", cloud.region = "us-east-1". OTel defines "semantic conventions" — standardized attribute names — so that tooling can interpret them consistently. Applications can also add their own custom attributes (e.g. finops.cost_center = "team-alpha").

Span events

Timestamped log entries attached to a span. Think of them as "something interesting happened during this span." An event has a name, a timestamp, and its own set of attributes. For example, a span representing an HTTP request might have an event named cache.miss with an attribute cache.key = "user:42".

Answer to "Does the application need to emit an OTel event with attributes?": Yes — if you want to track unique, domain-specific data within each span (like cost center, tenant ID, or billing tier), the application must explicitly set those as span attributes or emit span events with attributes. The trace ID and basic span metadata are handled automatically by the OTel SDK, but custom business data requires instrumentation code.

Receivers accept data in various formats (OTLP, Jaeger, Prometheus, etc.). Processors transform the data — batching for efficiency, filtering out noise, enriching with metadata, sampling to reduce volume. Exporters send the processed data to one or more destinations. This fan-out capability is critical: a single Collector can feed data to AWS X-Ray, Datadog, and a FinOps platform simultaneously.

The recommended production pattern is Agent + Gateway (Pattern C): lightweight agent collectors run alongside each application (low-latency ingestion, local buffering), forwarding to a centralized gateway collector that handles cross-cutting concerns like sampling, attribute enrichment, and multi-backend export.

Common misconception: "There is only one Collector." In practice, organizations often run many Collector instances — one per pod/host (agents) plus one or more centralized gateways. The architecture is flexible and scalable.

7. Answering our five questions

Q1: Is there typically an OTel Collector configured to receive application data from each application?
Yes. In production, most organizations deploy at least one Collector layer. The most common pattern is an agent-mode Collector running as a sidecar or DaemonSet alongside each application, feeding into a gateway-mode Collector for centralized processing and export. The Collector handles batching, retries, enrichment, and multi-destination export — things you don't want each application doing individually.
Q2: Is there usually only one OTel Collector?
No — there are usually many. A typical Kubernetes deployment runs one agent Collector per node (as a DaemonSet) plus one or more gateway Collector instances. In a multi-cloud setup, each cloud environment typically has its own Collector fleet. The "one Collector" mental model only applies to the simplest dev/test setups.
Q3: What are the standard OTel Collectors in AWS, GCP, Azure?
See the table in Section 6. In short: AWS → ADOT (feeds X-Ray, CloudWatch), GCP → Google Cloud OTel Collector (feeds Cloud Trace, Cloud Monitoring with native OTLP), Azure → Azure Monitor OpenTelemetry Distro (feeds Application Insights, Azure Monitor). All are distributions of the upstream open-source OTel Collector with cloud-specific plugins.
Q4: If we want to inspect OTel data from a trace, can we integrate with the OTel Collector?
Yes — this is the recommended approach. The customer's existing Collector is configured to add our OTLP endpoint as an additional exporter. Their Collector fans out data to their existing backend (e.g. Datadog) and to us simultaneously. We receive a copy of the telemetry; we don't intercept or modify the existing flow. This is a configuration change on the customer's Collector — no code changes in their applications.
Q5: Or must we be the OTel Collector?
We can, but we don't have to. There are two viable architectures (see Section 8). We can either (a) be an OTLP receiver endpoint that the customer's Collector exports to, or (b) run our own Collector instance (as a downstream gateway) that receives data from their Collector, processes it for our needs, and feeds our backend. Option (a) is simpler; option (b) gives us more control over filtering and transformation. In neither case do we replace the customer's existing Collector.

8. FinOps integration patterns: how we plug in

As a FinOps company, we have two primary integration architectures for consuming OTel trace data from customers. Both preserve the customer's existing observability setup.

Option A — OTLP receiver endpoint (simpler)

We expose an OTLP-compatible endpoint (gRPC on port 4317, HTTP on port 4318). The customer adds our endpoint as an additional exporter in their existing Collector configuration. We receive spans, process them in our own backend, and correlate them with cost data.

FinOps-specific value: By receiving trace data, we can map individual request paths to cloud resource consumption. A span that shows cloud.region = us-east-1, cloud.platform = aws_lambda, and finops.cost_center = team-alpha lets us attribute the Lambda invocation cost to the specific team, service, and even API endpoint that triggered it.

What the customer needs to do

In either option, the customer change is minimal — they add a few lines to their Collector config:

exporters:
  otlp/existing-backend:
    endpoint: datadog-agent:4317

  otlp/finops:                           # ← new
    endpoint: finops.example.com:4317    # ← new
    headers:                              # ← new
      api-key: "${FINOPS_API_KEY}"       # ← new

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/existing-backend, otlp/finops]  # ← added

9. Decision matrix

Consideration Option A: OTLP endpoint Option B: Our Collector
Setup complexity for us Lower — build an OTLP-compatible ingestion API Higher — deploy + operate a Collector fleet
Setup complexity for customer Same (add exporter config) Same (add exporter config)
Filtering / transformation Done in our application code Done in Collector config (more flexible, standard)
Data volume control Customer controls sampling We can add our own sampling layer
Protocol flexibility OTLP only Can also accept Jaeger, Zipkin, Prometheus
Best for MVP / quick integration Production scale / multi-tenant
Recommendation: Start with Option A (OTLP receiver endpoint) for the initial integration. It's the fastest path to receiving trace data and proving value. Migrate to Option B (our own Collector) as we scale and need finer-grained control over filtering, sampling, and multi-tenant processing.
